Client API — DDD + Mock

Un client API typé, défini comme interface dans le domaine et injecté dans le controller. L'implémentation mock est swappable contre un vrai client HTTP sans toucher à une seule ligne de logique applicative.

Client mock

Les données viennent d'un MockApiClient entièrement en mémoire. Les controllers ne savent pas d'où viennent les données — ils appellent leurs services respectifs et reçoivent des résultats paginés.

  • — Données statiques en mémoire — zéro latence réseau, zéro dépendance externe
  • — Recherche Pokémon filtrée et pagination calculées in-memory
  • — Deux services injectés séparément : IPokemonService et IBerryService
  • — Chaque controller ignore que l'autre resource existe — couplage minimal

Architecture

L'interface IApiClient vit dans le domaine — elle décrit ce que l'application peut demander, pas comment c'est livré. Deux services (IPokemonService, IBerryService) sont injectés séparément dans chaque controller via provide/inject.

Domain

domain/

Entités, interfaces IApiClient, IPokemonService, IBerryService. TypeScript pur — zéro import externe. Définit le contrat que toute implémentation doit respecter.

Infrastructure

infrastructure/

MockApiClient et HttpApiClient implémentent IApiClient. DTOs et mappers dans des sous-dossiers par ressource (pokemon/, berry/).

Controller

controller/

usePokemons() et useBerries() orchestrent state réactif et pagination. Chacun injecte uniquement son service — ignorant de l'autre.

Presentation

presentation/

ApiClientRoot fournit les services via provide/inject. PokemonList et BerryList consomment leurs controllers respectifs sans connaître le client.

Substituabilité du client

Passer du mock à PokéAPI n'a nécessité que des fichiers infrastructure : DTOs + mappers (Anti-Corruption Layer) dans pokemon/ et berry/, plus les deux clients assembleurs. Controllers, présentation et domaine sont restés inchangés — c'est le principe d'inversion de dépendances (DIP) appliqué à la couche transport.
